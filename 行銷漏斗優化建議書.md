# 月島甜點店 - 行銷漏斗優化建議書

**專案名稱**：MOON MOON 月島甜點店 - Island Landing  
**分析日期**：2026年1月28日  
**分析範圍**：導覽頁面內容流程、購物方式、商業邏輯、行銷漏斗設計

---

## 執行摘要

本建議書針對月島甜點店導覽網站進行全面分析，發現專案在**品牌故事與體驗設計**方面表現優秀，但在**行銷漏斗轉換**環節存在明顯斷層。主要問題為「情緒體驗」與「實際購買」之間的連結不夠緊密，導致用戶流失率可能偏高。

**核心發現**：
- ✅ 品牌定位清晰，情緒化行銷策略有效
- ✅ 技術架構完善，數據追蹤機制健全
- ⚠️ 轉換路徑不清晰，狀態選擇與購買流程脫節
- ⚠️ 付款流程依賴外部平台，無法有效追蹤
- ⚠️ 會員價值未充分發揮

## 二、問題診斷

### 2.1 行銷漏斗斷層 ⚠️ **高優先級**

**問題描述**：
- Check-in 體驗與實際購買流程完全分離
- 用戶選擇狀態後，沒有直接引導到對應商品
- 處方建議是隨機生成，與實際菜單無關聯

**影響**：
- 用戶完成狀態選擇後，不知道下一步該做什麼
- 情緒體驗無法轉化為實際購買行為
- 轉換率可能低於預期

**現有流程**：
```
選擇狀態 → 獲得處方建議 → [斷層] → 需要自己去找菜單 → 購買
```

**理想流程**：
```
選擇狀態 → 獲得處方建議 → 直接顯示對應商品 → 一鍵加入購物車 → 購買
```

### 2.2 轉換路徑不清晰 ⚠️ **高優先級**

**問題描述**：
- Hero 區三個入口都是外部連結，用戶離開網站
- 用戶可能不知道可以在網站直接購買
- 「ISLAND MENU」入口視覺層級較低，容易被忽略

**影響**：
- 流量分散到外部平台，無法集中轉換
- 用戶可能在其他平台完成購買，網站無法追蹤

### 2.3 購物車體驗不足 ⚠️ **中優先級**

**問題描述**：
- 購物車只在底部顯示，無法查看詳細內容
- 無法在購物車中修改數量
- 沒有商品圖片預覽

**影響**：
- 用戶可能忘記已選商品
- 修改購物車需要重新進入菜單，增加操作步驟

### 2.4 付款流程風險 ⚠️ **高優先級**

**問題描述**：
- 付款在 LINE 完成，無法追蹤付款狀態
- 用戶可能忘記付款或忘記回傳後五碼
- 沒有付款提醒機制

**影響**：
- 訂單完成率可能偏低
- 需要人工追蹤付款狀態，增加營運成本
- 用戶體驗不佳

### 2.5 會員價值未發揮 ⚠️ **中優先級**

**問題描述**：
- 登入與否的差異不明顯
- MBTI 資料未用於個性化推薦
- 沒有會員專屬優惠或累積機制

**影響**：
- 用戶缺乏登入動機
- 無法建立長期客戶關係
- 會員數據未充分利用

### 2.6 社交證明不足 ⚠️ **低優先級**

**問題描述**：
- 沒有客戶評價或推薦
- 沒有「熱門商品」標示
- 缺少「限時優惠」等促銷元素

**影響**：
- 新用戶缺乏購買信心
- 無法引導用戶選擇熱門商品

### 2.7 內容行銷整合不足 ⚠️ **低優先級**

**問題描述**：
- Spotify、桌布等內容與購買流程脫節
- 沒有引導用戶從內容到購買的橋樑

**影響**：
- 內容行銷投入無法轉化為銷售

## 三、優化建議

### 3.1 行銷漏斗優化方案

#### 方案 A：狀態選擇後直接引導購買 ⭐ **推薦優先實作**

**實作方式**：
1. 在狀態選擇結果卡片中，增加「立即購買此處方」按鈕
2. 點擊後自動將推薦商品加入購物車
3. 自動打開菜單 Modal，並高亮顯示已選商品
4. 顯示「已為您加入購物車，立即結帳？」提示

**預期效果**：
- 轉換率提升 30-50%
- 減少用戶操作步驟
- 強化情緒體驗與購買的連結

**技術實作要點**：
```typescript
// 在 result-card 中增加
const handleBuyPrescription = () => {
  // 解析 recommendation 字串，找到對應商品
  const [drink, dessert] = recommendation.split(' / ');
  
  // 從 menuCategories 中找到對應商品
  const drinkItem = findItemByName(drink);
  const dessertItem = findItemByName(dessert);
  
  // 自動加入購物車（使用預設規格）
  if (drinkItem) addToCart(drinkItem.name, drinkItem.defaultSpec, drinkItem.defaultPrice);
  if (dessertItem) addToCart(dessertItem.name, dessertItem.defaultSpec, dessertItem.defaultPrice);
  
  // 打開菜單並滾動到已選商品
  setShowMenu(true);
  // 高亮顯示已選商品
};
```

#### 方案 B：優化 Hero 區入口設計

**實作方式**：
1. 在 Hero 區增加「立即選購」按鈕（主要 CTA）
2. 三個外部連結改為次要按鈕樣式
3. 增加「本週熱銷」或「季節限定」標籤

**預期效果**：
- 提升網站內轉換率
- 減少流量外流

#### 方案 C：建立狀態與商品的映射關係

**實作方式**：
1. 在資料庫中建立 `state_prescriptions` 表
2. 每個狀態對應固定的商品組合（而非隨機）
3. 根據用戶選擇的狀態，精準推薦對應商品

**資料結構建議**：
```sql
CREATE TABLE state_prescriptions (
  id UUID PRIMARY KEY,
  state_key TEXT NOT NULL, -- 'calm', 'anxious', etc.
  drink_name TEXT NOT NULL,
  dessert_name TEXT NOT NULL,
  mission_text TEXT,
  advice_text TEXT,
  sort_order INTEGER
);
```

### 3.2 購物車體驗優化

**實作方式**：
1. 點擊底部購物車欄，打開詳細購物車 Modal

**預期效果**：
- 提升用戶對購物車的掌控感
- 減少因忘記已選商品而放棄購買的情況

#### 方案 B：增加商品縮圖預覽

**實作方式**：
1. 在購物車中顯示商品縮圖
2. 點擊縮圖可查看大圖
3. 在結帳確認 Modal 中也顯示商品圖片

### 3.3 付款流程優化

#### 方案 A：整合 LINE Pay API ⭐ **推薦優先實作**

**實作方式**：
1. 申請 LINE Pay 商家帳號
2. 整合 LINE Pay SDK
3. 在結帳確認頁面增加「LINE Pay 付款」按鈕
4. 付款成功後自動更新訂單狀態

**預期效果**：
- 提升付款完成率
- 減少人工追蹤成本
- 改善用戶體驗

**替代方案**：如果無法整合 LINE Pay，建議：
1. 增加付款狀態查詢功能
2. 自動發送付款提醒（透過 LINE Bot）
3. 在訂單確認頁面顯示「付款倒數計時器」

#### 方案 B：增加訂單追蹤功能

**實作方式**：

## 六、風險評估與應對

### 6.1 技術風險

**風險**：整合 LINE Pay API 可能遇到技術障礙  
**應對**：先實作替代方案（付款提醒系統），再逐步整合

**風險**：狀態與商品映射需要大量資料整理  
**應對**：先建立基本映射，再根據數據優化

### 6.2 營運風險

**風險**：自動化推薦可能導致某些商品過度銷售  
**應對**：設定庫存上限，自動切換推薦商品

**風險**：會員優惠可能影響利潤  
**應對**：計算 ROI，確保優惠帶來的增量收入大於成本

### 6.3 用戶體驗風險

**風險**：過度引導可能讓用戶感到被推銷  
**應對**：保持品牌調性，以「建議」而非「強迫」的方式引導
